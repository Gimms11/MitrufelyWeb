El ecosistema MitrufelyWeb está estructurado como una plataforma de tres capas. Cuenta con un backend transaccional en FastAPI, un frontend interactivo en React 19 y un microservicio independiente encargado de simular de manera asíncrona la entrega de pedidos físicos (Delivery). Para garantizar que la integración del software sea estable ante despliegues continuos, se estructuraron diferentes metodologías de testing que se detallan a continuación.

1. Pruebas de Software

La estrategia de aseguramiento de calidad adoptada en MitrufelyWeb cubre múltiples dimensiones funcionales y lógicas. El testing de software se divide estructuralmente en pruebas funcionales y pruebas unitarias puras, garantizando que el comportamiento lógico y el flujo transaccional se mantengan íntegros.

i. Pruebas Funcionales

Las pruebas funcionales se encargan de validar que los flujos transaccionales del negocio, los contratos de comunicación API HTTP y las integraciones entre servicios operen exactamente según las especificaciones técnicas.

A. Resumen Comparativo (Antes vs Después)

Antes (Estado Inicial)

  • Backend Integración: Cobertura de base de datos exitosa con pruebas de rollback transaccional ante fallos en checkout.
  • Microservicio de Delivery: Cobertura nula (0 tests). Riesgo alto de fallos de red silenciosos al invocar el webhook de finalización.
  • Error no controlado (Bug en main.py): Si el simulador asíncrono fallaba con una venta inexistente, el bloque except crasheaba con KeyError: 9999.
  • Red de Frontend: Dependencia de mocks internos de Axios. Sin validación de peticiones de red reales en el entorno de pruebas.
  • Flujos E2E: Ausencia de tests funcionales de extremo a extremo controlando un navegador web real.

Después (Estado Mitigado)

  • Backend Integración: Se mantuvo el 100% de los casos transaccionales de control de lotes FEFO sin regresiones.
  • Delivery robusto (+8 tests): Se implementó la suite completa que valida `/health`, `/deliveries` y la comunicación webhook con reintentos con backoff.
  • Bug Solucionado: Se corrigió el manejador de excepciones de main.py agregando if id_venta in deliveries:. El test de excepciones valida su estabilidad.
  • Simulación de red real (+9 tests MSW): Interceptación a nivel de red con MSW v2 de todas las peticiones a la API en Vitest.
  • Playwright E2E (+4 tests): Implementación de pruebas E2E en Chromium real para verificar pantallas de autenticación, flujos y redirecciones de seguridad.

B. Detalle de Implementación Técnica

Se crearon pruebas asíncronas en pytest para simular el comportamiento de entregas y el envío del webhook. Para simular fallos de red del backend principal y comprobar la resiliencia de la cola de reintentos con backoff exponencial, se mockeó el cliente HTTP:

@pytest.mark.asyncio
async def test_notify_backend_retry_then_success():
    with patch("httpx.AsyncClient.post", new_callable=AsyncMock) as mock_post:
        # Falla primero (500), tiene éxito en la segunda (200)
        mock_post.side_effect = [Response(500), Response(200)]
        
        with patch("asyncio.sleep", new_callable=AsyncMock) as mock_sleep:
            await _notify_backend(101)
            
            assert mock_post.call_count == 2
            mock_sleep.assert_called_once_with(1)  # Primer retraso de 2^0 = 1s

ii. Pruebas Unitarias

Las pruebas unitarias aíslan el software de dependencias externas (bases de datos físicas, llamadas de red reales o almacenamiento) para comprobar la validez de los algoritmos lógicos y las transiciones de estado del sistema.

A. Resumen Comparativo (Antes vs Después)

Antes (Estado Inicial)

  • Backend Unitario: Cobertura fuerte con 138 tests aislados (Auth, Ventas, Stock, SweetCoins). Módulos de reviews y dashboard sin cobertura dedicada.
  • Frontend Unitario: Cobertura nula (0 tests). Riesgo alto de inconsistencias de nombres de variables tipadas.
  • Tareas Periódicas Celery: Sin pruebas unitarias. Riesgo de errores de lógica silenciosos en los disparadores asíncronos.

Después (Estado Mitigado)

  • Backend Unitario robusto (+19 tests): Se integraron unit tests para DashboardService y ReviewService mockeando sesiones de base de datos.
  • Frontend Unitario (+8 tests): Se integró Vitest y se crearon tests de Zustand Store y Button render.
  • Celery Beat verificado (+10 tests): Implementación de pruebas unitarias sobre la orquestación, reintentos y configuraciones del planificador (Celery Beat).

📖 Glosario

📖 Glosario de Términos

Pruebas Funcionales (Functional Testing)
Metodología de pruebas orientada a validar el comportamiento operativo de las características del sistema contra las especificaciones del negocio (contratos de API, webhook, respuestas HTTP).
Pruebas Unitarias (Unit Testing)
Pruebas enfocadas en componentes individuales del software de forma totalmente aislada (servicios de lógica, stores de Zustand, componentes de UI), abstrayendo o mockeando cualquier interacción con bases de datos o red externa.
MSW (Mock Service Worker)
Librería que intercepta llamadas HTTP salientes a nivel de red en el entorno de desarrollo y pruebas de Node.js mediante Service Workers simulados, permitiendo probar código cliente con respuestas falsas sin alterar el cliente Axios.
Playwright
Framework moderno de código abierto para automatizar pruebas E2E (End-to-End) en navegadores reales (Chromium, Firefox, WebKit), permitiendo verificar la renderización visual, interacción de formularios y redireccionamientos.
Zustand
Gestor de estado global simplificado para React, utilizado en el frontend de MitrufelyWeb para gestionar de forma persistente la sesión del usuario (JWT) en sessionStorage.
Celery Beat
Planificador periódico de tareas distribuidas en segundo plano para Celery, programado para correr de forma automatizada (ej. expiración diaria de cupones, lotes y anulación de ventas antiguas).
Backoff Exponencial
Algoritmo de reintentos de red que incrementa progresivamente el tiempo de espera entre intentos fallidos de manera exponencial (por ejemplo, 1s, 2s, 4s, etc.), reduciendo la sobrecarga de solicitudes sobre un servidor inestable o congestionado.
NeonDB (PostgreSQL)
Base de datos relacional serverless utilizada por el backend FastAPI, cuya persistencia, transacciones, triggers y procedimientos almacenados (como sp_expirar_lotes_vencidos) garantizan la consistencia FEFO del inventario.
