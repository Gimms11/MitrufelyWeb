# Plan completo de optimización de rendimiento FrontEnd — MitrufelyWeb

## Objetivo
Llevar las 3 páginas (`/`, `/dashboard`, `/catalog/admin`) al 100 en Lighthouse (Performance, Accessibility, Best-Practices, SEO) auditando contra el build de producción servido por nginx, no el dev server.

## Hallazgos base (de la auditoría)
- **Lighthouse se corrió contra el dev server** (`:5173`, sin minificar) → números catastróficos no reflejan producción.
- **22 MB de PNGs sin optimizar** en `public/` (5 de ellos = 17 MB, el `4.png` de 2.4 MB es el LCP del Hero).
- **Sin `manualChunks`** → vendor duplicación, chunk principal de 392 KB.
- **Sin compresión** (ni gzip ni brotli) ni servidor de producción (sin nginx/Dockerfile).
- **`@import` de Google Fonts bloqueante** en CSS (14 pesos de 2 familias).
- **Google Identity script cargado globalmente** pero NO se usa (login usa OAuth redirect).
- **`AdminLayout` devuelve `null`** durante auth check → flash/CLS.
- **framer-motion en critical path** del Hero (animaciones on-mount bloquean LCP).
- **Imágenes sin `width`/`height`** → CLS.
- **Dependencias muertas**: `jspdf` (350KB) + `exceljs` (900KB) sin uso.

## FASE 1 — Optimización de imágenes (mayor impacto en LCP)

### 1.1 Convertir PNGs a WebP/AVIF + reducir tamaño
Para cada imagen grande de `public/`, generar versiones WebP (y AVIF si es viable) a resolución de display:
- `4.png` (1874×839, 2.4MB, LCP Hero) → `4.webp` (~1200px, ~80KB)
- `8.png` (2315×1317, 4.8MB) → `8.webp` (~1200px, ~100KB)
- `9.png` (1861×1841, 4.0MB) → `9.webp` (~1000px, ~90KB)
- `10.png` (1858×1778, 4.2MB) → `10.webp` (~1000px, ~90KB)
- `image.png` (1.6MB) → evaluar si se usa; el comentario en HeroSection lo menciona pero el código usa `4.png`
- `fbb5dddd-*.png` (348KB + 228KB) → WebP
- Los 6 JPEGs mal etiquetados como .png (`pack_*`, `truffle_*`, `hero_truffles`) → recomprimir

**Herramienta**: usar un script con `sharp` (dependencia dev temporal) o conversión manual con `cwebp`. Se mantiene el PNG original como fallback.

### 1.2 Eliminar imágenes huérfanas
Borrar de `public/` las no referenciadas en `src/`: `image.png`, `hero_truffles.png`, `pack_detalle.png`, `pack_estudio.png`, `pack_premium.png`, `truffle_custom.png`, `truffle_sweetcoins.png` (~5 MB). Renombrar `fbb5dddd-*.png` (nombres con espacios/paréntesis = hazard de URL).

### 1.3 Actualizar referencias en el código
- `HeroSection.tsx:66`: cambiar `background-image: url(/4.png)` → `<img src="/4.webp" fetchpriority="high" decoding="async" className="absolute inset-0 h-full w-full object-cover object-right" />` con fallback PNG.
- `AboutHeroSection.tsx`, `SharedMomentsSection.tsx`, `BenefitsSection.tsx`: usar `<picture>` con source WebP + img fallback, añadir `loading="lazy"` (excepto hero), `width`/`height`.

### 1.4 Preload del LCP en `index.html`
```html
<link rel="preload" href="/4.webp" as="image" type="image/webp" fetchpriority="high" />
```

## FASE 2 — Configuración de build de Vite

### 2.1 Añadir `manualChunks` en `vite.config.ts`
```ts
build: {
  target: 'es2022',
  sourcemap: mode !== 'production',
  chunkSizeWarningLimit: 1000,
  rollupOptions: {
    output: {
      manualChunks: {
        'react-vendor': ['react', 'react-dom', 'react-router'],
        'query-vendor': ['@tanstack/react-query', '@tanstack/react-table'],
        'form-vendor': ['react-hook-form', 'zod', '@hookform/resolvers'],
        'ui-vendor': ['framer-motion', 'lucide-react', 'sonner'],
        'chart-vendor': ['recharts'],
      },
    },
  },
},
```

### 2.2 Añadir plugin de compresión Brotli + Gzip
Instalar `vite-plugin-compression2` (devDep) y configurarlo para generar `.br` y `.gz` de todos los assets JS/CSS/SVG.

### 2.3 Eliminar dependencias muertas
Quitar `jspdf` y `exceljs` de `package.json`. Mover `zustand` de devDependencies a dependencies (es runtime). Quitar `@types/canvas-confetti` de dependencies → devDependencies.

### 2.4 Añadir `rollup-plugin-visualizer` (devDep)
Script `"analyze": "ANALYZE=true vite build"` para inspeccionar el bundle y validar mejoras.

## FASE 3 — Resource hints y fonts (`index.html` + `index.css`)

### 3.1 Mover Google Fonts de `@import` CSS a `<link>` en HTML
Eliminar la línea 2 de `index.css` (`@import url('https://fonts.googleapis.com/...')`). En `index.html` añadir:
```html
<link rel="preconnect" href="https://fonts.googleapis.com" />
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin />
<link rel="preload" as="style" href="https://fonts.googleapis.com/css2?family=Inter:wght@400;500;600;700;800&family=Outfit:wght@600;700;800;900&display=swap" />
<link rel="stylesheet" href="https://fonts.googleapis.com/css2?family=Inter:wght@400;500;600;700;800&family=Outfit:wght@600;700;800;900&display=swap" />
```
Reducción de 14 pesos a 8 (eliminar 300 de Inter y 300/400/500 de Outfit que no se usan).

### 3.2 Eliminar Google Identity script global
Quitar de `index.html` la línea `<script src="https://accounts.google.com/gsi/client" async defer></script>`. No se usa (login usa OAuth redirect manual). Borrar también `src/types/google.d.ts`.

### 3.3 Mejorar meta tags SEO/accesibilidad
- `lang="es"` (la app es española)
- `<title>Mitrufely — Trufería artesanal premium</title>`
- `<meta name="description" content="...">`
- `<meta name="theme-color" content="#5c0f1b">`
- `<meta name="color-scheme" content="light">`

## FASE 4 — Optimización de código (TBT y CLS)

### 4.1 Code-split recharts dentro del dashboard
Extraer los gráficos de `AdminDashboardPage.tsx` y `DashboardPage.tsx` a un componente `DashboardCharts.tsx` cargado con `lazy()`. Esto separa recharts (1.17MB) del chunk de la página y solo carga cuando hay datos.

### 4.2 Reemplazar framer-motion del Hero con CSS
En `HeroSection.tsx`, reemplazar las animaciones `motion` on-mount (`initial="hidden" animate="show"` con stagger) por animaciones CSS con `@keyframes` y `animation-delay`. Esto saca framer-motion del critical path del `/`. Mantener framer-motion donde hay interacciones reales (modales, dropdowns).

### 4.3 Hacer `AdminLayout` lazy + skeleton en auth gate
- En `router.tsx:6`, cambiar `import AdminLayout from ...` (eager) → `lazy(() => import(...))`.
- En `AdminLayout.tsx:48-50`, reemplazar `return null` por `return <PageLoader />` para evitar flash/CLS.

### 4.4 Añadir dimensiones a imágenes (CLS)
Revisar todos los `<img>` y añadir `width`/`height` explícitos. Especialmente en `CatalogAdminPage.tsx` (imágenes de producto 48×48), `BenefitsSection.tsx`, `AboutHeroSection.tsx`.

### 4.5 Devtools condicional con import dinámico
En `providers.tsx`, asegurar que `ReactQueryDevtools` no colisione en el bundle de producción usando import dinámico condicional.

## FASE 5 — Servidor de producción (nginx + Docker)

### 5.1 Crear `Dockerfile` multi-stage en `_frontEnd/`
```dockerfile
# Stage 1: build
FROM node:22-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: serve
FROM nginx:alpine
COPY --from=build /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
```

### 5.2 Crear `nginx.conf` con:
- **Brotli/Gzip**: servir `.br`/`.gz` pre-comprimidos (generados por el plugin de Vite) con `brotli_static on` / `gzip_static on`.
- **Cache headers inmutables** para `/assets/*` (hash en filename) → `Cache-Control: public, max-age=31536000, immutable`.
- **Cache corto** para `index.html` → `Cache-Control: no-cache`.
- **Cache medio** para imágenes → `max-age=86400`.
- **SPA fallback** → `try_files $uri $uri/ /index.html`.
- **Tipos MIME** correctos para `.webp`, `.avif`.

### 5.3 Añadir servicio frontend al `docker-compose.yml` raíz
```yaml
frontend:
  build: ./_frontEnd
  ports:
    - "5173:80"   # o el puerto que prefieras
  depends_on:
    - api
```

## FASE 6 — Verificación y auditoría final

### 6.1 Build + servir producción
```bash
cd _frontEnd
npm run build          # genera dist/ con chunks optimizados + .br/.gz
npm run preview        # o docker-compose up frontend
```

### 6.2 Auditar con Lighthouse (modo correcto)
Ejecutar en Chrome **incógnito**, sin extensiones, con throttling móvil simulado:
```bash
npx lighthouse http://localhost:4173/ --output html --output-path report-home.html --preset=desktop
npx lighthouse http://localhost:4173/dashboard --output html --output-path report-dashboard.html --preset=desktop
npx lighthouse http://localhost:4173/catalog/admin --output html --output-path report-catalog.html --preset=desktop
```
Para auditar `/dashboard` y `/catalog/admin` hay que estar autenticado: usar el flag `--extra-headers` con un JWT o auditar manualmente tras login en el navegador.

### 6.3 Validar con bundle analyzer
```bash
npm run analyze        # abre el visualizador del bundle
```
Confirmar que:
- El chunk principal bajó de 392 KB → ~150 KB
- `chart-vendor` está aislado
- No quedan duplicados de framer-motion en muchos chunks
- Las imágenes WebP pesan <150KB cada una

### 6.4 Checklist de verificación
- [ ] LCP < 2.5s en las 3 páginas
- [ ] FCP < 1.8s
- [ ] CLS < 0.1 (idealmente 0)
- [ ] TBT < 200ms
- [ ] Performance ≥ 90 (objetivo 100)
- [ ] Accessibility ≥ 95
- [ ] SEO = 100
- [ ] Best-Practices = 100
- [ ] Las imágenes se sirven como WebP con fallback
- [ ] Brotli/Gzip activo (verificar headers `Content-Encoding: br`)
- [ ] Cache headers inmutables en `/assets/*`

## Orden de ejecución recomendado
1. **FASE 2** (config Vite + deps) — base técnica, rápido
2. **FASE 1** (imágenes) — mayor impacto en LCP
3. **FASE 3** (fonts + index.html) — impacto en FCP
4. **FASE 4** (código) — impacto en TBT/CLS
5. **FASE 5** (nginx + Docker) — servidor de producción
6. **FASE 6** (verificación) — medir resultado final

## Notas
- La conversión de imágenes puede hacerse con un script de Node usando `sharp` (instalado temporalmente como devDep) o manualmente con herramientas CLI.
- Si `image.png` y las `pack_*.png`/`truffle_*.png` se usan vía backend (URLs configuradas), se conservarán pero optimizadas; si no, se eliminan.
- El React Compiler ya está activo (babel-plugin-react-compiler) — eso ayuda al rendimiento de render, no requiere cambios.
